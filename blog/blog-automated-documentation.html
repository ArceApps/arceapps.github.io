Documentación Automática para Proyectos Android: CHANGELOG.md y Más Allá 🚀

Descubre cómo automatizar completamente la documentación de tu proyecto Android con GitHub Actions: desde CHANGELOG.md hasta documentación de API en vivo con Kotlin/KDoc

La documentación manual es el enemigo silencioso de todo desarrollador. Siempre prometemos que "la actualizaremos después", pero esa promesa rara vez se cumple. ¿Y si te dijera que puedes tener documentación siempre actualizada, changelogs automáticos y API docs en vivo sin esfuerzo manual?

En este artículo vamos más allá de los conventional commits y el versionado semántico que ya conoces, para crear un ecosistema completo de documentación automática orientado específicamente a proyectos Android con Kotlin.

🎯 Lo que aprenderás en este artículo:

  • CHANGELOG.md automático con semantic-release avanzado
  • Documentación de API con Dokka para Kotlin
  • GitHub Pages deployment automático
  • Pipeline completo que combina todo en GitHub Actions
  • Ejemplos específicos para proyectos Android

🏗️ Anatomía del Pipeline de Documentación Perfecta

Nuestro objetivo es crear un flujo que, con cada commit a main, automáticamente:

📝 Analiza commits → 🔍 Extrae cambios → 📚 Genera docs → 🚀 Publica en GitHub Pages

Estructura del Proyecto Android

Partimos de un proyecto Android típico con algunas adiciones para automatización:

my-android-app/
├── app/                     # Código Android
│   ├── src/main/kotlin/
│   └── build.gradle.kts     # Configuración Dokka
├── docs/                    # Documentación generada (auto-creada)
├── .github/
│   └── workflows/
│       └── docs.yml         # Pipeline automatización
├── release.config.js        # Semantic-release config
├── package.json             # Dependencias Node para tooling
└── CHANGELOG.md             # Auto-generado

⚙️ Configuración Base: Semantic Release para Android

Empezamos configurando semantic-release adaptado específicamente para proyectos Android. Este va más allá del setup básico:

package.json para Tooling

{
  "name": "my-android-app-docs",
  "version": "1.0.0",
  "private": true,
  "devDependencies": {
    "@semantic-release/changelog": "^6.0.3",
    "@semantic-release/git": "^10.0.1",
    "@semantic-release/github": "^9.2.6",
    "@semantic-release/commit-analyzer": "^11.1.0",
    "@semantic-release/release-notes-generator": "^12.1.0",
    "semantic-release": "^22.0.12"
  },
  "scripts": {
    "semantic-release": "semantic-release"
  }
}

release.config.js Avanzado para Android

module.exports = {
  branches: ["main", "develop"],
  plugins: [
    // 1. Analizar commits convencionales
    ["@semantic-release/commit-analyzer", {
      preset: "angular",
      releaseRules: [
        { type: "feat", section: "🚀 Nuevas Funcionalidades", release: "minor" },
        { type: "fix", section: "🐛 Correcciones", release: "patch" },
        { type: "perf", section: "⚡ Optimizaciones", release: "patch" },
        { type: "refactor", section: "♻️ Refactoring", release: "patch" },
        { type: "docs", section: "📚 Documentación", release: false },
        { type: "style", section: "💄 Estilos", release: false },
        { type: "test", section: "🧪 Testing", release: false },
        // Reglas específicas para Android
        { type: "ui", section: "🎨 Interfaz de Usuario", release: "minor" },
        { type: "gradle", section: "🔧 Configuración", release: "patch" },
        { scope: "deps", section: "📦 Dependencias", release: "patch" }
      ]
    }],
    
    // 2. Generar release notes personalizadas
    ["@semantic-release/release-notes-generator", {
      preset: "angular",
      presetConfig: {
        types: [
          { type: "feat", section: "🚀 Nuevas Funcionalidades" },
          { type: "fix", section: "🐛 Correcciones" },
          { type: "perf", section: "⚡ Optimizaciones de Rendimiento" },
          { type: "refactor", section: "♻️ Refactoring" },
          { type: "ui", section: "🎨 Mejoras de UI/UX" },
          { type: "gradle", section: "🔧 Configuración y Build" },
          { type: "security", section: "🔒 Seguridad" },
          { type: "deps", section: "📦 Actualización de Dependencias" }
        ]
      }
    }],

    // 3. Generar CHANGELOG.md
    ["@semantic-release/changelog", {
      changelogFile: "CHANGELOG.md",
      changelogTitle: "# Changelog\n\nTodos los cambios notables de este proyecto se documentarán automáticamente en este archivo.\n\nEl formato está basado en [Keep a Changelog](https://keepachangelog.com/es/1.0.0/),\ny este proyecto adhiere a [Semantic Versioning](https://semver.org/lang/es/)."
    }],

    // 4. Commit de archivos actualizados
    ["@semantic-release/git", {
      assets: [
        "CHANGELOG.md", 
        "app/build.gradle.kts",  // Si actualizas versionName
        "gradle.properties"      // Si usas properties para versión
      ],
      message: "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
    }],

    // 5. Crear GitHub Release
    "@semantic-release/github"
  ]
};

📚 Documentación de API con Dokka para Kotlin

Dokka es la herramienta oficial de JetBrains para generar documentación de Kotlin. Es como Javadoc, pero mucho más poderosa:

Configuración en build.gradle.kts

plugins {
    id("org.jetbrains.dokka") version "1.9.10"
    // ... otros plugins
}

// Configuración de Dokka
dokka {
    outputDirectory.set(file("../docs/api"))
    
    dokkaSourceSets {
        named("main") {
            // Configuración del módulo principal
            moduleName.set("MyAndroidApp API")
            moduleVersion.set(findProperty("VERSION_NAME").toString())
            
            // Incluir solo archivos públicos
            documentedVisibilities.set(
                setOf(
                    org.jetbrains.dokka.DokkaConfiguration.Visibility.PUBLIC,
                    org.jetbrains.dokka.DokkaConfiguration.Visibility.PROTECTED
                )
            )
            
            // Enlaces a documentación externa
            externalDocumentationLink {
                url.set(URL("https://developer.android.com/reference/"))
                packageListUrl.set(URL("https://developer.android.com/reference/androidx/package-list"))
            }
            
            // Configurar samples
            samples.from("src/main/kotlin/samples/")
            
            // Suprimir archivos generados
            suppressedFiles.from(
                "src/main/kotlin/**/BuildConfig.kt",
                "src/main/kotlin/**/*_*.kt"  // Archivos generados por Room, etc.
            )
        }
    }
}

🚀 Pipeline Completo de GitHub Actions

Ahora creamos el workflow que orquesta todo el proceso. Este pipeline es específicamente diseñado para proyectos Android:

.github/workflows/docs.yml

name: 📚 Documentation & Release Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

env:
  JAVA_VERSION: 17
  GRADLE_VERSION: wrapper

jobs:
  # Job 1: Análisis y Release Semántico
  semantic-release:
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    outputs:
      new-release-published: ${{ steps.semantic.outputs.new-release-published }}
      new-release-version: ${{ steps.semantic.outputs.new-release-version }}
    
    steps:
      - name: 🔄 Checkout Repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Necesario para semantic-release
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: 📱 Setup Android Environment  
        uses: actions/setup-java@v4
        with:
          java-version: ${{ env.JAVA_VERSION }}
          distribution: 'temurin'

      - name: 🟢 Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: 📦 Install Dependencies
        run: npm ci

      - name: 🏷️ Run Semantic Release
        id: semantic
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: npx semantic-release

  # Job 2: Generar Documentación de API
  generate-docs:
    runs-on: ubuntu-latest
    needs: semantic-release
    if: always() # Ejecutar incluso si no hay release
    
    steps:
      - name: 🔄 Checkout Repository
        uses: actions/checkout@v4
        with:
          ref: main  # Asegurar que tenemos la última versión

      - name: 📱 Setup Android Environment
        uses: actions/setup-java@v4
        with:
          java-version: ${{ env.JAVA_VERSION }}
          distribution: 'temurin'

      - name: 🗃️ Cache Gradle Dependencies
        uses: actions/cache@v4
        with:
          path: |
            ~/.gradle/caches
            ~/.gradle/wrapper
            ~/.android/build-cache
          key: ${{ runner.os }}-gradle-${{ hashFiles('**/*.gradle*', '**/gradle-wrapper.properties') }}
          restore-keys: |
            ${{ runner.os }}-gradle-

      - name: 🔧 Grant Execute Permission for Gradlew
        run: chmod +x gradlew

      - name: 📚 Generate Dokka Documentation
        run: ./gradlew dokkaHtml --no-daemon

      - name: 📝 Create Documentation Index
        run: |
          mkdir -p docs
          
          # Crear index.html principal
          cat > docs/index.html << 'EOF'
          
          
          
            MyAndroidApp - Documentación
            
            
          
          
            
🚀 MyAndroidApp - Documentación

            
            

              
📚 Documentación de API

              
Explora la documentación completa generada automáticamente desde el código Kotlin.

              Ver Documentación de API →
            


            

              
📋 Changelog

              
Historial completo de cambios, generado automáticamente desde commits convencionales.

              Ver Changelog →
            


            

              
🏷️ Releases

              
Todas las versiones publicadas con notas de lanzamiento detalladas.

              Ver Releases →
            


            

              Última actualización: $(date -u '+%Y-%m-%d %H:%M UTC')
            

          
          
          EOF

      - name: 🎨 Enhance API Documentation  
        run: |
          # Verificar si la documentación se generó
          if [ -d "app/build/dokka/html" ]; then
            # Copiar documentación generada
            cp -r app/build/dokka/html/* docs/api/ || echo "Error copying docs"
            
            # Personalizar CSS (si existe)
            if [ -f "docs/api/styles/style.css" ]; then
              echo "
              :root { --color-primary: #6366f1; }
              .library-name a { color: var(--color-primary) !important; }
              " >> docs/api/styles/style.css
            fi
          else
            echo "❌ No se generó documentación Dokka"
            ls -la app/build/dokka/ || echo "Directorio no existe"
          fi

      - name: 📤 Upload Documentation Artifacts
        uses: actions/upload-artifact@v4
        with:
          name: documentation
          path: docs/
          retention-days: 30

  # Job 3: Deploy a GitHub Pages
  deploy-docs:
    runs-on: ubuntu-latest
    needs: [semantic-release, generate-docs]
    if: github.ref == 'refs/heads/main'
    
    # Permisos necesarios para GitHub Pages
    permissions:
      contents: read
      pages: write
      id-token: write

    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - name: 📥 Download Documentation
        uses: actions/download-artifact@v4
        with:
          name: documentation
          path: docs/

      - name: 🔧 Setup Pages
        uses: actions/configure-pages@v4

      - name: 📤 Upload to Pages
        uses: actions/upload-pages-artifact@v3
        with:
          path: docs/

      - name: 🚀 Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

      - name: 📨 Notification
        if: needs.semantic-release.outputs.new-release-published == 'true'
        run: |
          echo "🎉 Nueva versión ${{ needs.semantic-release.outputs.new-release-version }} publicada!"
          echo "📚 Documentación actualizada en: ${{ steps.deployment.outputs.page_url }}"

2. Configuración Gradle Avanzada

Para proyectos grandes, puedes configurar Dokka por módulos:

// En el build.gradle.kts del proyecto raíz
plugins {
    id("org.jetbrains.dokka") version "1.9.10" apply false
}

subprojects {
    apply(plugin = "org.jetbrains.dokka")
    
    tasks.withType().configureEach {
        dokkaSourceSets {
            named("main") {
                // Configuración específica por módulo
                when (project.name) {
                    "app" -> {
                        moduleName.set("App Principal")
                        includes.from("Module.md")
                    }
                    "core" -> {
                        moduleName.set("Core - Utilidades")
                        includes.from("Core.md")
                    }
                    "data" -> {
                        moduleName.set("Data Layer")
                        includes.from("Data.md")
                    }
                }
            }
        }
    }
}

📊 Monitoreo y Métricas

Para proyectos enterprise, puedes añadir métricas sobre tu documentación:

- name: 📈 Documentation Metrics
  run: |
    # Contar líneas de documentación
    KDOC_LINES=$(find app/src -name "*.kt" -exec grep -c "^\s*/\*\*" {} \; | awk '{s+=$1} END {print s}')
    
    # Contar clases documentadas
    DOCUMENTED_CLASSES=$(find app/src -name "*.kt" -exec grep -l "/\*\*.*class" {} \; | wc -l)
    
    # Crear badge dinámico
    echo "📚 Líneas de documentación: $KDOC_LINES" >> $GITHUB_STEP_SUMMARY
    echo "🏛️ Clases documentadas: $DOCUMENTED_CLASSES" >> $GITHUB_STEP_SUMMARY

💡 Consejos Pro y Troubleshooting

Optimizaciones de Performance

  • Cache inteligente: El pipeline usa cache de Gradle para builds más rápidos
  • Builds condicionales: Solo regenera docs si hay cambios en código
  • Paralelización: Jobs independientes ejecutan en paralelo

Debugging Común

# Si Dokka falla, verificar:
1. ./gradlew dokkaHtml --stacktrace --info
2. Revisar configuración de visibilidad en dokka
3. Verificar que no hay KDoc malformado
